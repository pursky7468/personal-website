The LINE Bot outfit recommender is a side project: feed it the day's weather and a wardrobe inventory, and it suggests what to wear. Small scope, clear inputs, predictable outputs — exactly the kind of project where an LLM feels like the obvious shortcut. I didn't take it. Here's why — and where that reasoning breaks down.

## The appeal is real

An LLM recommendation would handle edge cases automatically. No need to
enumerate every possible outfit combination. Natural language output. Easy to
extend to new clothing categories. The model does the hard work.

This is genuinely compelling. I understand why people default to it.

## The specific shape of this problem

The core question for any component: how do you verify it works?

For a rule-based recommendation engine, the answer is straightforward:

```csharp
// Given: 12°C, rainy, casual occasion
// Expected: jacket + waterproof layer + closed shoes
Assert.Equal(expectedOutfit, recommender.Suggest(weather, wardrobe, occasion));
```

132 passing tests. Deterministic. Fast.

I should be honest about what this testing approach assumes: that the correct answer is singular. For a simple weather-to-outfit mapping with a fixed wardrobe and explicit rules, that's defensible. The rules are small enough to enumerate, and the failure modes are visible.

For an LLM recommendation — even a well-constrained one with structured output and temperature=0 — the picture changes:

- Behavior can shift silently across model versions without any code change
- The verification model shifts from assertions to evaluation: semantic scoring, schema validation, acceptance regions — valid approaches, but a fundamentally different engineering surface
- When the problem space is bounded enough to test exhaustively, that extra verification complexity buys nothing

This isn't a claim that LLM outputs can't be tested. It's a claim that for this specific bounded problem, the verification overhead wasn't worth it.

## The actual tradeoff

Using LLM would have made the failure modes *harder to audit* in exchange for *less code to write*. That's the honest tradeoff.

The other side deserves acknowledgment too: rule-based systems have their own costs. When rules accumulate, they produce implicit priority conflicts, hidden coupling, and maintenance surfaces that aren't obviously better than a well-constrained model. I didn't hit that wall here — the rule set stayed small — but it's a real risk on the other side of the scale.

## Where this reasoning breaks down

This decision made sense because the problem had a specific shape:

- Closed input space: fixed wardrobe, finite weather conditions
- Rules small enough to enumerate and test exhaustively
- Output correctness verifiable by inspection

Change any of those, and the tradeoff shifts. A system where correct answers are genuinely plural — or where the input includes natural language like *"I'm going to a business casual dinner but I walk a lot"* — is exactly where semantic interpretation earns its place. Rule systems don't handle fuzzy intent extraction gracefully. LLMs do.

The interesting architectural question isn't "LLM or rules?" It's where the boundary sits. For this project, the boundary was clear. That won't always be the case.
