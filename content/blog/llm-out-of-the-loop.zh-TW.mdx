LINE Bot 穿搭推薦機器人是一個 Side Project：輸入當天天氣和衣櫥清單，它會建議你穿什麼。範疇小、輸入明確、輸出可預測——這種專案恰好是讓人覺得 LLM 顯然是捷徑的那種。我沒走那條路。以下是原因——以及這種決策邏輯何時不再適用。

## 吸引力是真實存在的

LLM 推薦可以自動處理邊界情況。不需要列舉每種可能的穿搭組合。自然語言輸出。容易擴展到新的衣物類別。模型負責做困難的工作。

這確實很有說服力。我能理解為什麼大家會直接選擇它。

## 這個問題的具體特徵

任何元件的核心問題：你怎麼驗證它能正常運作？

對於規則式推薦引擎，答案很直接：

```csharp
// Given: 12°C, rainy, casual occasion
// Expected: jacket + waterproof layer + closed shoes
Assert.Equal(expectedOutfit, recommender.Suggest(weather, wardrobe, occasion));
```

132 個測試全數通過。確定性。快速。

坦白說，這種測試方法背後有一個前提假設：正確答案是唯一的。對於一個有固定衣櫥和明確規則的簡單天氣與穿搭對應關係，這個假設是成立的。規則小到可以全部列舉，失敗模式也清晰可見。

對於 LLM 推薦——即使是有結構化輸出和 temperature=0 的嚴格約束版本——情況就不同了：

- 行為可能在模型版本之間無聲地改變，而不需要任何程式碼變更
- 驗證模式從斷言（assertion）轉移到評估（evaluation）：語義評分、schema 驗證、可接受範圍——都是有效的方法，但在工程層面上根本是兩回事
- 當問題空間夠小、可以窮舉測試時，額外增加的驗證複雜度毫無實質好處

這不是在說 LLM 的輸出無法被測試，而是說：對於這個特定的有界問題，驗證成本不值得付出。

## 實際的取捨

使用 LLM 會讓失效模式*更難排查*，換取的是*更少的程式碼*。這才是誠實的取捨。

另一面也值得承認：規則式系統有它自己的成本。當規則累積，會產生隱含的優先衝突、隱性耦合，以及維護面——這些不一定比設計良好的受限模型更好。我在這個專案沒遇到那道牆，因為規則集保持精簡，但那是天平另一端真實存在的風險。

## 這種決策邏輯何時不再適用

這個決策之所以成立，是因為問題有特定的特徵：

- 封閉的輸入空間：固定衣櫥、有限的天氣條件
- 規則小到可以全部列舉並窮舉測試
- 輸出結果的正確性可透過人工檢視來驗證

改變其中任何一項，取捨就會移位。一個正確答案真正是多元的系統——或者輸入包含自然語言，像是「我要去一個商務休閒晚餐，但我需要走很多路」——正是語義解讀能發揮價值的地方。規則系統無法優雅地處理模糊的意圖提取，LLM 可以。

有趣的架構問題不是「LLM 還是規則？」，而是邊界在哪裡。在這個專案裡，邊界很清楚。不會每次都這樣。
