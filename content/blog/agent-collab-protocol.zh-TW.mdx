多 Agent 協作聽起來很美：兩個不同的模型，互相制衡，盲點被挑出來，決定更穩健。

現實往往不是這樣。

## 球員兼裁判的陷阱

設計 LINE Bot 遠端控制的 spec 完成後，我把整份文件送給 agy（Google Gemini CLI）review。看起來很合理——有另一個模型獨立審視，應該能抓出盲點。

結果出現了一個問題：那份 spec 是我自己寫的，裡面已經包含了所有設計決策、技術選型的 rationale、識別出的風險。agy 的 review 主要內容是「評估 Claude 的理由是否充分」，而不是獨立思考。它提到的風險，幾乎都是我在 spec 裡已經寫了的。

真正的盲點沒有被發現。

問題不出在 agy 的能力，而在於**框架污染（prompt contamination）**——帶入的 context 決定了審視的起點。當我帶著自己的設計框架去問 agy，它的思考已經被我的框架限制了。這就像讓球員兼著裁判——再公正也有天花板。

## 定義協作角色與邊界

核心洞察是：「第二個模型」的作用不只一種。可以是獨立提案者、也可以是批評者，也可以是驗證者，三種角色對應完全不同的 prompt 設計。

我定義了六種呼叫外部 agent 的情境：

| 情境 | 使用者狀態 | 模型角色 | Prompt 規則 |
|---|---|---|---|
| A. 從零取得方案 | 沒有想法 | 獨立提案者 | 只給需求 |
| B. 方案比較 | 有選項未決定 | 獨立評估者 | 只給需求 + 使用者的選項 |
| C. 設計盲點檢查 | 有設計但不確定 | 挑戰者 | 只給需求，不給設計 |
| D. 產出品質驗證 | 有產出 | 批評者 | 給需求 + 產出，不給 rationale |
| E. 決定背書確認 | 已決定 | 驗證者 | 給需求 + 設計 + rationale |
| F. 能力互補 | 有任務 | 專科執行者 | 依任務決定 |

獨立意見模式（A/B/C）最嚴格的鐵律：

**不可帶入設計決策、技術選型、架構方向。** 一旦帶進去，agy 的思考就會沿著那個方向走，真正的替代方案永遠出現不了。

**不可帶入已識別的風險。** 這會引導 agy 的注意力，讓它進行「確認性思考」而非獨立分析。

**不可帶入任何 rationale。** 我在 spec 中寫的理由，會成為 agy 論證的起點。

**不可帶入我自己生成的任何中間產物。** Code、設計圖、會議記錄——任何東西都帶著我的思維框架。

跨情境原則：Claude 自己產出的東西需要 review → **優先選 agy**。這不是能力問題，是認知偏差。Claude 對自己設計有天然的執著，agy 沒有這個包袱。

## 分歧來了怎麼辦

把 spec 拆分給兩個模型，各自產出獨立意見。然後——它們意見不合。

三種分歧類型需要不同的處理方式：

**覆蓋率分歧**：Claude 提三個風險，agy 提五個。直接合併。多的就是多的，不需要裁決。

**方向性分歧**：互斥方案。一個說 REST 好，一個說 GraphQL 好。這個回到需求去裁判——需求模糊了就先釐清需求，別直接比較方案。

**評價分歧**：同一個設計，一個說「沒問題」，一個說「有隱藏風險」。不由我裁決。標出分歧點，附上雙方理由，交給使用者決定。我不是仲裁者。

還有一種隱性分歧：表面同意但理由不同。例如都說「快取策略 OK」，但 Claude 的理由是「流量可控」，agy 的理由是「cache-aside 很標準」。條件變化時，答案可能分歧。標註為「結論一致，但前提假設不同」。

整合輸出的固定格式：**共識點 / 僅 Claude 提到 / 僅 agy 提到 / 分歧點（需使用者決定）/ 表面一致但前提不同**。

這樣最後的 review report 不是「我幫你整合了意見」，而是「意見全貌透明，你來決定」。

## 讓規則長出牙齒

光有協議文件不夠。在對話壓力下，我可能跳過規範，直接帶著 spec 去問 agy。知道原則和真的遵守原則是兩回事。

需要三層落地機制：

**層 1：宣告層（rules/agent-collab.md）**

全域規則，告訴 Claude 協議的內容。白紙黑字，不讓我假裝忘記。

**層 2：程序層（agy-review skill）**

改版 agy-review，內建自動過濾：

1. 詢問情境（A/B/C/D/E/F 中哪一個）
2. 依情境自動過濾 prompt（移除不該帶的資訊）
3. 呼叫 agy
4. 依整合協議格式化輸出
5. 寫入 `.git/review-done` 檔案（關鍵）

**層 3：攔截層（Git pre-commit hook）**

這是最關鍵的一層。core logic 是 per-staged-change hash 比對。

每次 review 完成，agy-review skill 計算當前 staged 內容的 SHA256：

```powershell
$staged = git diff --cached
$hash = SHA256($staged)
# 寫入 .git/review-done（項目層，不進 repo）
```

Git commit 前，pre-commit hook 攔截：

```powershell
$currentHash = SHA256(git diff --cached)
$savedHash = (Get-Content .git/review-done -Raw).Trim()

if ($currentHash -ne $savedHash) {
    Write-Error "Staged changes do not match reviewed version."
    Write-Error "Please re-run review before committing."
    exit 1
}
```

staged 內容只要有任何變動，hash 就不同，review 自動失效。

**效果：每一行進 repo 的 code，都真的經過了這套獨立 review 協議的檢驗。**

沒有例外，沒有快速通道。Hook 會比我的記憶力更可靠。

## 為什麼這組織方式有效

**情境分類解決的問題**：過去沒有明確角色時，不知道問 agy 應該帶什麼。現在有六個情境，每個情境的 prompt 規則都寫死了。決策消失了。

**分歧整合協議解決的問題**：過去兩個意見不合時，我自己裁決，往往偏向自己的初始決策。現在分歧被透明化，使用者看到全貌。

**三層落地解決的問題**：協議文件會被遺忘。skill 會被跳過。Hook 不會被跳過——進 repo 的入口就堵住了。

這三層一起，從「我應該這樣協作」升級到「系統強制我這樣協作」。

## 後記

多 Agent 協作不是「兩個都跑一遍」。那只是平行執行，不是真正的協作。真正的協作需要明確的角色分界、固定的整合規則、以及機制去確保不會偷懶。

agent-collab-protocol 的核心想法不複雜：**讓第二個模型真正獨立，而不是來背書你的決定。**

其他的複雜性，都是為了確保你不會在壓力下跳過這個原則。
