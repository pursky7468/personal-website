## The First Error Report

The message was brief: "Device not found, but it looks fine in Device Manager."

That sentence is itself a puzzle. Device Manager showing normal means Windows recognizes the device, the driver loaded successfully, no yellow exclamation marks. But the application says Device not found. Those two things shouldn't both be true — unless the way the app "finds" the device is different from what Device Manager checks.

The attached screenshot confirmed: the camera controller (Computar series lens, controlled via LensConnect) showed no errors in Device Manager. darkFlash DN-D.exe launched and immediately threw an error dialog, stopping at the initialization stage.

I plugged in the same device locally, launched the same version of the software: everything worked.

This is the worst kind of bug: **reproducible, but only on the client's machine.**

---

## Starting with Event Log

The first step in remote diagnosis is always Event Log.

I asked the client to open `eventvwr.msc`, filter the System log, narrow the time window to the minute when the software launched. Two entries appeared:

```
Source: Application Error
Event ID: 1000
Faulting application: DN-D.exe
Exception code: 0xc0000005
```

0xc0000005 is Access Violation — the process failed to read or write at a memory address. This isn't the typical symptom of "device not found." It looks more like the code obtained a NULL handle and then tried to use it anyway.

The second entry was more interesting:

```
Source: Microsoft-Windows-DriverFrameworks-UserMode
Event ID: 10110
A problem has occurred with your device driver software.
Device: USB\VID_10C4&PID_EA80
```

VID `10C4` is Silicon Labs. PID `EA80` is the CP2112 — a USB HID to SMBus bridge chip, the target device for the SLABHIDtoSMBus driver.

This is not the camera or lens VID/PID. This is the communication bridge layer in between.

---

## Tracing the Instance ID

I asked the client to open Device Manager with "View → Devices by connection," find the CP2112, right-click Properties → Details → Device Instance ID.

The returned string was a USB path in the format:

```
USB\VID_10C4&PID_EA80\{instance_hash}
```

At the same time, I asked them to check hidden devices (View → Show hidden devices) and look for ghost devices — the same VID/PID appearing more than once, with one carrying a yellow exclamation mark but hidden.

Result: **two CP2112 instances — one active, one hidden with Code 45** (Device was not migrated due to partial or ambiguous match).

Code 45 typically appears when a device was previously connected to a different USB port, Windows left the old device record, and the new connection didn't clean up the stale record. This alone shouldn't crash an application — unless the application opens the device using `SetupDiEnumDeviceInterfaces` to enumerate all interfaces matching a GUID, takes the first one it finds, and that first one happens to be the dead hidden instance.

---

## Following the Handle with PID

At this point I needed to confirm: which handle did the software actually get?

I asked the client to open `procexp64.exe` (Sysinternals Process Explorer) just before the crash, find the DN-D.exe process, open Handle view (Ctrl+H), and filter for `\Device\`.

The returned handle list contained:

```
\Device\{pdo_path}
```

The PDO path format pointed to the Physical Device Object of the dead hidden instance — not the active one. Confirmed: **the software enumerated the zombie device.**

Next, right-click the hidden instance in Device Manager → Uninstall device (check "Delete the driver software for this device"), replug the USB to let Windows re-enumerate.

Relaunch the software.

Clean startup.

---

## Why It Didn't Happen Locally

The local machine was a fresh installation — that device had never been connected to a different USB port. The client's machine had been in use for over two years, with one USB hub change and one workstation relocation in between. Every time the port changed, Windows left behind a stale device record. At some point, the enumeration order happened to put the dead instance first.

This is a known issue with Windows device enumeration: `SetupDiEnumDeviceInterfaces` does not guarantee that results are ordered by "most recently used" or "active first" — it follows the internal registry order. Driver authors should enumerate all interfaces, attempt to open each one, and move on if it fails, rather than assuming the first result is valid.

## Systematic Troubleshooting Reference

The following covers the general approach for "Device Manager shows normal but the application can't find the device" problems.

### 1. Identify the Failure Layer

| Symptom | Likely Layer |
|---|---|
| Yellow exclamation in Device Manager | Driver layer |
| Device Manager clean, software can't find device | Application-layer enumeration |
| Device Manager clean, software crashes | Handle misuse (usually NULL handle) |

### 2. Event Log Key Filters

- **System log**: DriverFrameworks-UserMode (Event ID 10110)
- **Application log**: Exception code 0xc0000005 (Access Violation)
- Time window: lock to the minute when the software launched

### 3. Finding Ghost Devices

```
Device Manager → View → Show hidden devices
```

Same VID/PID appearing multiple times:

- Code 45: device changed ports, stale record not cleaned up
- Code 43: driver initialization failure
- Code 28: driver not found

### 4. Tracing the Handle

Use Process Explorer (Handle view) to confirm that the `\Device\` path points to the dead instance, not the active one.

### 5. Clearing Ghost Devices

Select the hidden dead instance → Uninstall device → check delete driver → replug the device.

For batch removal across multiple machines:

```powershell
# Requires devcon.exe (included with Windows Driver Kit)
devcon remove "@USB\VID_10C4&PID_EA80\{old_instance_id}"
```

### 6. Preventive Recommendation (for Driver/App Developers)

Don't assume the first interface returned by `SetupDiEnumDeviceInterfaces` is active:

```cpp
// Correct: enumerate all, attempt open, continue on failure
while (SetupDiEnumDeviceInterfaces(...)) {
    HANDLE h = CreateFile(detail->DevicePath, ...);
    if (h != INVALID_HANDLE_VALUE) {
        // Use this handle
        break;
    }
    // Continue to next interface
}
```

## Epilogue: DeviceDoctor

After this investigation wrapped up, I spent a few days wrapping the process above into a small tool — DeviceDoctor. Give it a VID/PID, and it automatically lists all active and hidden instances, flags Code 45/43/28, and suggests which ones to remove. The tool itself isn't complex, but with it, the next time this problem comes up, the "chase the ghost device" step goes from 15 minutes to 30 seconds. Sometimes the most useful tools aren't ones that solve new problems — they're the ones that stop known problems from wasting time.
