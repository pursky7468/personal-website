在整合視覺系統（camera）和機械臂的過程中，我遇到了一個看起來很簡單，但實際上很危險的 bug：座標轉換的向量約定不一致。這個問題不在於數學本身，而在於跨越不同領域慣例的邊界時，很容易寫出看起來正確、但語義上完全錯誤的程式碼。

## 背景：整合視覺和機械臂系統

現在的專案需要：
- 相機拍攝工作台上的物體
- 檢測物體位置和姿態
- 指示機械臂去拿取

這涉及三個座標系：
- **Camera frame (c)** — 相機的視角
- **Robot base frame (r)** — 機械臂基座
- **Arm end-effector frame (a)** — 夾爪

```
向量轉換路徑：

  [Arm (a)] ──r_T_a──► [Robot (r)] ──c_T_r──► [Camera (c)]
      └──────────────────── c_T_a ───────────────────►

  箭頭方向 = 向量的轉換方向（從 source frame 到 destination frame）
```

## Column Vector Convention（標準機器人學約定）

機器人學（Craig、Spong 教科書）用 column vector convention：

向量為列向量，transform matrix **左乘**向量：

```
v' = T * v
```

命名約定是 `{destination frame}_T_{source frame}`，例如 `c_T_r` 表示「把 robot frame 的向量帶到 camera frame」的 transform。

Frame chaining 的規則（cancellation rule）：

```
c_v = c_T_r * r_T_a * a_v
```

消去規則：當兩個矩陣相鄰，**左矩陣的 source frame = 右矩陣的 destination frame**，中間的 frame 消去：

```
  c_T_r  *  r_T_a
  ───┬──    ──┬──
     └── r ──┘   ← 相鄰的 r 消去
         │
         ▼
       c_T_a
```

任意長度的 chain 依序消去：

```
  c_T_r  *  r_T_b  *  b_T_a
  ───┬──    ──┬─┬──   ──┬──
     └── r ──┘ └── b ──┘
                 │
                 ▼
               c_T_a
```

## Row Vector Convention（圖形學和 OpenTK）

OpenTK（C#/.NET 圖形數學庫）用 row vector convention：

向量為行向量，transform matrix **右乘**向量：

```
v' = v * T
```

這導致 frame chaining 的乘法順序和 column convention **完全相反**。

| 動作 | Column convention | Row convention (OpenTK) |
|------|-------------------|-------------------------|
| 向量應用 transform | `v' = T * v` | `v' = v * T` |
| Frame chaining（同樣命名） | `C_T_A = C_T_B * B_T_A` | `C_T_A = B_T_A * C_T_B` |

Row convention 的消去規則方向**相反**：**左矩陣的 destination frame = 右矩陣的 source frame** 時消去：

```
  B_T_A  *  C_T_B    ← row convention（同樣的 {dest}_T_{src} 命名）
  ─┬───     ─┬───
   └─── B ───┘   ← 這次是左矩陣的 dest 對右矩陣的 src
        │
        ▼
      C_T_A
```

消去方向反轉的根本原因：若 `M_row = M_col^T`，轉置代數保證

```
  column:  A_col * B_col
                ↓ 兩側取轉置
  row:     B_row * A_row   （順序自動反轉）

  因為 (A * B)^T = B^T * A^T
```

所以在 row convention 中寫 frame chain，乘法順序與 column convention 的**視覺上完全相反**，但表達的幾何意義相同。

## 重要區分：Vector Convention vs Memory Layout

這是這個領域最常見的混淆，值得單獨說清楚。

| 概念 | 定義 | 影響 |
|------|------|------|
| **Row/column vector convention** | 向量是 row 還是 column，決定 transform 左乘還是右乘 | 乘法語義、乘法順序、frame chaining 方向 |
| **Row/column major (memory layout)** | 矩陣元素在記憶體中的排列順序（哪個 index 連續存放） | 記憶體存取效率，**不影響乘法語義** |

兩者沒有必然關聯。Row vector convention 的庫不一定用 row-major storage，反之亦然。

OpenTK 剛好同時使用兩者，但造成「乘法順序差異」和「rotation 需要轉置」的原因只有 **row vector convention**，不是 memory layout。如果有人說「OpenTK 需要轉置是因為 row-major」，他混淆了這兩件事。

```
row-major storage:  控制 M[0][1] 在記憶體中的位置
row vector conv.:   決定 v' = v * M 還是 v' = M * v  ← 這個才是重點
```

## 問題根源：混淆向量約定

問題程式碼：

```csharp
var c_T_a = c_T_r * r_T_c;
```

**正確版本應為：**

```csharp
// Column convention 語義（在 OpenTK 中需透過 helper 處理）
var c_T_a = c_T_r * r_T_a;
```

原始程式碼的兩個問題：

1. **`r_T_c` 語義錯誤**：按命名約定，`r_T_c` 是「把 camera frame 的向量帶到 robot frame」，也就是 `c_T_r` 的 inverse，根本不是 `r_T_a`。用 cancellation rule 驗：`c_T_r * r_T_c` → r 消去 → 結果是 `c_T_c`（identity，不是我們要的 `c_T_a`）

2. **OpenTK 是 row convention，乘法順序必須反轉**：即使把 `r_T_c` 改成正確的 `r_T_a`，直接用 `*` 運算子在 OpenTK 中計算，也是錯的

## 實際程式碼範例

### 錯誤版本（混合約定）

```csharp
// OpenTK = row convention，但用 robotics naming
Matrix4 c_T_r = GetCameraToRobotTransform();
Matrix4 r_T_c = GetRobotToCameraTransform();  // 語義上是 c_T_r 的 inverse！

// 兩個錯誤疊加：
// 1. r_T_c 語義錯（cancellation rule: c_T_r * r_T_c → c_T_c，不是 c_T_a）
// 2. row convention 下乘法順序也錯
var c_T_a = c_T_r * r_T_c;
```

### 正確版本（使用公司的 Column Vector Helper）

```csharp
public static class ColumnVectorTransforms
{
    /// <summary>
    /// 在 OpenTK row convention 環境中建立 column convention 語義的 Matrix4。
    /// OpenTK 用 row vector convention（v' = v * M），其中 axis vectors 是 rows。
    /// Column convention 的旋轉矩陣 R 其 axis vectors 是 columns，
    /// 放入 OpenTK 時需要轉置為 R^T，使 v' = v * R^T 等價於 v' = R * v。
    /// </summary>
    public static Matrix4 Create(Vector3 translation, Matrix3 rotation)
    {
        return new Matrix4(rotation.Transpose(), translation);
    }

    /// <summary>
    /// 按 column convention 語義合成兩個 transforms：Compose(A, B) = A * B。
    /// 數學原理：column convention 的 A_col * B_col，在 row convention 中等價於
    /// B_row * A_row，因為 (A_col * B_col)^T = B_col^T * A_col^T = B_row * A_row。
    /// 轉置關係使乘法順序自動反轉，所以實作為 rhs * lhs。
    /// </summary>
    public static Matrix4 Compose(Matrix4 lhs, Matrix4 rhs)
    {
        // 對應 column convention: lhs * rhs
        return rhs * lhs;
    }
}

// 使用方式
var c_T_r = ColumnVectorTransforms.Create(trans_cam, rot_cam);
var r_T_a = ColumnVectorTransforms.Create(trans_arm, rot_arm);

// c_T_a = c_T_r * r_T_a（column convention 語義）
// Cancellation rule 驗證：c_T_r * r_T_a → r 消去 → c_T_a ✓
var c_T_a = ColumnVectorTransforms.Compose(c_T_r, r_T_a);
```

## 為什麼這個 Bug 難發現

- **編譯器不報錯**：Matrix4 乘法語法完全合法
- **結果「有點對」**：旋轉角度小的時候誤差不明顯
- **混合約定太常見**：圖形學和機器人學的庫幾乎必然同時使用
- **命名讓人放鬆警覺**：`c_T_r` 這種 robotics naming 讓人以為數學也是 column convention

## 解決策略

**1. 清晰邊界層**

一個 module 或 static class 專門負責兩種約定的橋接，所有 robotics coordinate 運算都經過它。這個 class 的存在本身就是一個信號：「這裡我們用 column convention 的思維」。

**2. 命名和 Cancellation Rule 作為第一道防線**

每次寫 transform chain，先用 cancellation rule 驗：`c_T_r * r_T_a` → r 消去 → 得到 `c_T_a`，符合預期才繼續。一旦驗不出來（例如 `c_T_r * r_T_c` → 得到 `c_T_c`，不對），就是有問題。

**3. Frame Chaining 測試**

```csharp
[Fact]
public void FrameChaining_ColumnConvention_CorrectResult()
{
    // 三個座標系，已知相對位置
    var c_T_r = ColumnVectorTransforms.Create(new Vector3(1, 0, 0), Matrix3.Identity);
    var r_T_a = ColumnVectorTransforms.Create(new Vector3(0, 1, 0), Matrix3.Identity);

    var c_T_a = ColumnVectorTransforms.Compose(c_T_r, r_T_a);

    // arm 在 camera frame 的位置應為 (1, 1, 0)
    Assert.Equal(new Vector3(1, 1, 0), c_T_a.ExtractTranslation());
}
```

不只測單個 transform，驗 frame chaining 的合成結果才能發現這類 bug。

## 結語

向量約定不是抽象的數學細節。跨越不同領域的庫時，它成為很實際的工程問題。

**關鍵經驗：**

- **Row-major storage 和 row vector convention 是兩件不同的事**：前者是記憶體佈局，後者是數學約定，後者才是乘法順序差異的根本原因
- **Cancellation rule 是最快的自我驗證工具**：每次寫 transform chain 就驗一次，錯誤在鍵盤上就能攔截
- **在混合約定的邊界上建立清晰、可測試的轉換層**，比聰明的 one-liner 更重要

把約定明確化，bug 自己會浮出來。
